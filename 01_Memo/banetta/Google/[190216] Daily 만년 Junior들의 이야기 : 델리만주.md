해커톤을 왜 해야하죠?
- 협업을 미리 경험해 볼 수 있다.
  개발자는 혼자서 할 수 있는 직업이 아니기 때문에 미리 경험해 볼 좋은 기회
- 포트폴리오와 개발 경험을 쌓기에 유리하다.
  수상을 못한다고 해도 프로젝트는 생성되기 때문에 자소서나 이력서를 작성할 때 유리하게 작용할 수 있음
- 개발에 열정적인 사람을 찾기 쉽다.
  학교에서는 찾기 힘든 열정적인(?)사람이 많이 없다고 생각함, 해커톤에는 이미 열정 가득한 사람이 많기 때문에 정보 교류도 유리하고 네트워킹도 해두면 아주 좋음
- 해커톤을 나가서 아무것도 못해도 터닝포인트가 될 수 있기에 좋은 경험이 될 수도 있음

##  고학점 대신 개발자 커뮤니티를 선택한 이야기 - 김종찬 (NAVER) [A Hall]

어떻게 커뮤니티를 만났나?
  - 2011 - Android Developer Night : 개발자들이 자유롭게 모여서 경험 공유를 하는 것을 본 경험
  - 2012 - Google 대학생 해커톤 참여 : 가장 어렸지만, 내로라하는 학교의 박사와 같은 참가자로서의 경험, 해커톤이라는 재미있는 형태의 이벤트를 만났고, 즐거운 경쟁을 경험
  - 2012 - 학생 GDG 커뮤니티 리딩 시작 : 학생들이 학교라는 틀을 벗어나 자신을 더 열리고 개방적인 환경에서 성장하게 하기 이제 막 시작하는 1학년 새내기였지만, 팀을 리딩하고 커뮤니티를 꾸려나가는 경험 수행

-   생산적인 일에 대하여 도전정신을 가지는게 좋은 싲가을 만들어 냈따.
-   당장 크고 직접적인 이득을 주는 경험치가 아니라고 생각되더라도 시작하는 단계라면 적극적으로 임하고 스스로 기회를 만들어내기
-   학생으로서 리더로서의 역할을 수행하는 것은 이르면 이를수록, 많으면 많을수록 좋다
-   주변에 항상 함께하는 사람들이 내가 배울만한 사람들이고 같이 성장하는 사람들로 채워져 있을 때, 내가 원하는 나의 수준을 이루기가 가장 빠른 방법이다.

어떻게 커뮤니티를 활용했나?
- 가장 최신의 기술 트랜드와 도구 파악 (아키텍쳐, 라이브러리등)
- 현업 개발자들에게 듣는 좋은 개발자의 기준 알기
- 현업 개발자분들의 경험과정을 최적화 시키기
- 현업 개발자를 만나는것도 좋지만 언제나 같은 세대, 또래 개발자들과 많이 아는것이 중요
- 커뮤니티를 운영하기 위해서는 소수의 사람들로는 어렵다. 기술적으로도 운영적으로도 많은 동료들이 필요함
- (중요) 그렇게 자주 만나고, 엮이는 학생 동료들끼리 자신의 기술력을 키우기위한 도전거리들을 공유하고 같이 도전 할 수 있음 (SW Maestro, 삼성 SW 멤버쉽 등)

어떤 성장을 이루어냈나?
-   좋은 코드를 짜는 법을 일찍 알고 실천 : 다른 학생들, 현업의 개발자들로부터 내 코드를 비교하며 문제를 파악하고 경험을 공유받았습니다.
-   새로운 라이브러리등 기술의 트랜드를 빠르게 내 프로젝트에 적용 : 보통 회사에서는 새로운 트랜드를 쉽게 적용하고 받아들이기 쉽지 않을 수 있습니다. 학생시절에 그런 접근을 쉽게 해보고, 이슈를 겪어보는 것도 좋은 성장 요소.
-   빠르게 성장하는 큰 회사/스타트업 인턴 경험 : 커뮤니티에서 기술을 주제로 사람들을 이끄는 일을 함으로서, 현 회사의 인턴 수행 군생활 이후 개발자 커뮤니티를 같이 운영하시던 현업 개발자 분으로부터 인턴 오퍼 제공받음
-   현업에서 일어나는 기술 수준을 파악하고 자신의 경쟁력을 갖추기 위한 성장 목표 수립 : 현업이라고 언제나 최고의 기술 수준을 다루지 않습니다. 현업의 평균적인 기술 수준 대비 내 실력을 어디까지 끌어올려야 짧은 시간내에 자신이 성장 할 수 있는 기업이나 프로그램에 몸 담을 수 있는지 파악하고 이를 수련함.

### 웹페이지 열심히 만들었는데 다른 사람한테도 보이게 하려면 어떻게 하지?

클라이언트 vs 서버
-   클라이언트는 스마트폰, 크롬, 익스플로러다.
-   서버는 데이터센터, 클라우드다.
-   서버를 구성하는 요소 3가지
    -   컴퓨터
    -   WAS(Web Application Server) : 웹프레임워크들
    -   DB
-   클라이언트 (by. Front-end Dev.)
    -   눈에 보이는 부분(UI)
    -   브라우저에서 동작
    -   HTML, CSS, JavaScript
-   서버 (by. Back-end Dev.)
    -   안 보이는 부분
    -   인증 로직 처리
    -   데이터베이스 접근
정적 웹사이트 vs 동적 웹사이트 (정적 웹사이트를 위주로 다룸)
-   정적 웹사이트는 WAS와 DB없이 동작
-   동적 웹사이트는 로직을 처리하고 데이터를 저장한다. (로직은 WAS 담당, 데이터는 DB담당)
-   정적웹사이트는 only file, 동적 웹사이트는 file, logic. data. etc...

Github vs AWS vs Google
-   Github
    -   무료
    -   커스텀 도메인 가능
    -   HTTPS 지원
    -   Git Push 배포
    -   초기 배포 대기 없음
    -   동적 페이지 연계 불가
-   AWS
    -   약간의 과금
    -   커스텀 도메인 가능
    -   https 지원
    -   gui 배포
    -   초기 배포 대기 있음
    -   동적 페이지 연계 가능
-   Google
    -   약간의 과금
    -   커스텀 도메인 가능
    -   https 지원
    -   CLI 배포
    -   초기 배포 대기 없음
    -   동적 페이지 연계 가능
